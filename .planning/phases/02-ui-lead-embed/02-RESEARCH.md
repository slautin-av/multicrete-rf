## Summary

Этап 2 — чистый UI-слой поверх готового детерминированного движка `calc()`. Три участка несут реальный риск, остальное — переиспользование уже существующего кода лендинга.

1. **Доставка письма заявки.** PHP `mail()` с shared-хостинга Reg.ru на Яндекс — капризна. Главные грабли: `From` на чужом домене (`@yandex.ru`) → письмо отбраковывается как спуфинг; порт 25 на Reg.ru заблокирован (нужен 587/TLS); инъекция заголовков через перевод строки в полях. Рекомендация: `From` строго на собственном домене сайта (мультикрит.рф), Яндекс — в `Reply-To`; жёсткая защита от header-injection; реальная проверка доставки перед сдачей; готовый fallback на Web3Forms.

2. **Политика 152-ФЗ.** Отдельная статическая страница по типовому шаблону (через проверенный конструктор), 8 обязательных разделов, реквизиты оператора — плейсхолдерами под подстановку владельцем. Галочка согласия `required` со ссылкой на страницу.

3. **Адаптив большого модального окна.** Полноэкранный/прокручиваемый оверлей на телефоне: `position: fixed; inset: 0`, новые единицы `dvh`, прокрутка тела окна с блокировкой фона (`overflow:hidden` на `body` + `overscroll-behavior: contain`), безопасные зоны iOS, закрытие по Esc/фону, доступность.

**Primary recommendation:** Собрать `calculator/` одним статическим модулем в тёмной теме лендинга; UI вызывает `calc(inputs, PARAMS)` и рисует `breakdown`; воронка — рендерить «шапку»+оговорки сразу, детализацию держать в скрытом блоке и раскрывать после валидной отправки формы. PHP-обработчик писать с `From` на собственном домене + защитой от инъекций + параметром «источник»; обязательно проверить живую доставку и держать Web3Forms наготове как fallback.

| Рендер калькулятора (поля, результат) | Browser / Client (JS) | — | Расчёт детерминирован, идёт в браузере (`calc()` — чистая функция, без сети) |

| Приём и пересылка заявки | API / Backend (PHP на Reg.ru) | Web3Forms (fallback) | `mail()` шлёт письмо владельцу; серверная валидация и защита от инъекций — на этом тире |

### Pattern 1: Контракт движка (что UI рисует)
**What:** UI вызывает чистую функцию и раскладывает поля результата. НЕ пересчитывает ничего сам.
```js
// Source: calculator/engine.js (этап 1, VERIFIED grep в коде)
import { calc } from './engine.js';
import { PARAMS } from './params.js';

const breakdown = calc({ S: 12, T: 5, klass: 2 }, PARAMS);

// «Шапка» (показывать СРАЗУ, без контактов — D-09):
//   breakdown.trudozatraty, .vspomMaterialy, .nakladnye, .pribyl, .nds,
//   .transport (=0), .komandirovki (=0), .materialy, .itogo, .perM2
// Оговорки (СРАЗУ, D-11): breakdown.ogovorki  // массив из 3 строк
// Детализация (за контактами — D-10):
//   .stDrobestruika, .stObespylivanie, .stNanesenie, .stElastomer,
//   .stGruntPredv, .stGruntOsn, .stOchistitel, .sumWorks, .stITR, .esn,
//   .sklad, .garantiya, .rabotyItogo, .massElastomer, .cenaElastomer ...
```
**Важно:** галочка «>80 кг» (UI-02) НЕ передаётся в `calc()` и НЕ меняет число — движок всегда считает деталь до 80 кг (CALC-07). Чекбокс только меняет показываемый текст оговорки (`PARAMS.tekstOgovorki80kg` уже зашит в `ogovorki[0]`). Это честно: окончательную цену для тяжёлой детали считают отдельно. `[VERIFIED: calculator/engine.js строки 117-141]`

### Pattern 2: Переиспользование модального окна лендинга
**What:** Не изобретать оверлей. Берём готовую механику.
```js
// Source: landing.html строки 932-934, 951 (VERIFIED)
function openModal()  { document.getElementById('modal-overlay').classList.add('open');  document.body.style.overflow = 'hidden'; }
function closeModal() { document.getElementById('modal-overlay').classList.remove('open'); document.body.style.overflow = ''; }
function handleOverlay(e){ if (e.target === document.getElementById('modal-overlay')) closeModal(); }
document.addEventListener('keydown', e => { if (e.key === 'Escape') closeModal(); });
```
Для калькулятора нужен ОТДЕЛЬНЫЙ оверлей (`#calc-modal-overlay`), т.к. окно лендинга — это «Заказать звонок» с другой разметкой. Стиль `.modal-overlay`/`.modal` уже есть и работает в тёмной теме (`landing.html` строки 355-363, 410-411). `[VERIFIED: landing.html]`

### Pattern 3: Воронка «детализация за контакты»
**What:** Детализация в DOM есть всегда, но скрыта; раскрывается после валидной отправки формы.
```js
// Псевдокод (D-10). Детализация НЕ секрет — это раскладка, не цена,
// поэтому достаточно клиентского show/hide, серверная защита не нужна.
detailsBlock.hidden = true;            // по умолчанию скрыто
form.addEventListener('submit', async (e) => {
  e.preventDefault();
  if (!form.checkValidity()) return;   // имя/телефон/e-mail + согласие required
  await sendLead(formData);            // POST на send-lead.php (или Web3Forms)
  detailsBlock.hidden = false;         // награда: полная раскладка
});
```

### Anti-Patterns to Avoid
- **Дублировать коэффициенты/формулы в UI.** Всё число — из `calc()`/`PARAMS`. UI только форматирует. (CALC-09 — один источник истины.)
- **Менять `engine.js`/`params.js`.** Они вне границ этапа 2 (CONTEXT: правки движка — Deferred). Самопроверка `engine.selftest.js` сломается.
- **Прятать ИТОГО/оговорки за форму.** Нарушает LEAD-01/UI-04/D-11. Цена и контекст — бесплатно.
- **`From: ...@yandex.ru` в PHP-письме.** Главная причина попадания в спам (см. Скользкий участок 1).
- **Класть `req.body` напрямую в заголовки письма.** Header injection (см. Скользкий участок 1).
- **Хранить Web3Forms `access_key` / любые токены прямо в JS-коде, попадающем в репозиторий.** Правило CLAUDE.md: секреты выносим. `access_key` Web3Forms — публичный по дизайну (виден в форме), но не коммитить лишнего; PHP-обработчик секретов в коде не должен содержать вовсе.

## Don't Hand-Roll

| Проблема | Не строить | Использовать | Почему |
|----------|------------|--------------|--------|
| Текст политики 152-ФЗ с нуля | Самописный юр-текст | Конструктор (privacy-generator.ru / Tilda / els24) → подставить реквизиты | 8 обязательных разделов, формулировки под Роскомнадзор; ошибка = штраф |
| Расчёт стоимости в UI | Повторение формул Excel | `calc()` из этапа 1 | Уже выверено число-в-число с эталоном, есть selftest |
| Оверлей/модалка | Новый механизм окна | `.modal-overlay`/`openModal` лендинга | Готово, в тёмной теме, D-08 |
| Парсинг/санитизация полей письма | «Просто подставить в mail()» | preg_match-фильтр CRLF + явные заголовки | Header injection — известная дыра PHP `mail()` |
| Маска/валидация телефона | Сложная regex-маска | HTML5 `type="tel"` + `required` + лёгкая JS-проверка | B2B-форма, перфекционизм не нужен; главное — не блокировать отправку |

**Key insight:** Этап на 80% — сборка из готовых блоков (движок, модалка, стили, шаблон политики). Самописать стоит только: PHP-обработчик (с готовой защитой от инъекций), рендер `breakdown`, адаптив окна.

## Скользкий участок 1 — Доставка письма заявки (LEAD-02, D-01/D-02/D-03)

### Рекомендованный подход: PHP `mail()` с корректными заголовками

**Грабли №1 (главные) — `From` на чужом домене.** Если в письме `From: MultiCrete@yandex.ru` (адрес-получатель), Яндекс видит, что письмо отправлено НЕ с серверов Яндекса от имени яндекс-адреса → спуфинг → спам/отказ. У домена сайта (мультикрит.рф) нет SPF/DKIM, разрешающего слать «от яндекса». `[CITED: namecheap/jesandy — DMARC alignment с видимым From]`

**Правило:** `From` — на СОБСТВЕННОМ домене хостинга/сайта (напр. `noreply@мультикрит.рф` или `site@<домен-на-reg.ru>`), Яндекс-адрес владельца — в `Reply-To` (чтобы «Ответить» уходило владельцу). Тогда SPF домена сайта на Reg.ru с большей вероятностью проходит. `[CITED: Reg.ru help — «From не должен быть @mail.ru/@yandex.ru»]`

**Грабли №2 — порт 25 заблокирован на Reg.ru.** На виртуальном хостинге Reg.ru исходящие на порт 25 закрыты; sendmail/SMTP должен использовать **порт 587 с TLS**. Для голого `mail()` это вопрос конфигурации хостинга (sendmail_path / php.ini), не кода — но если письма не уходят вовсе, причина часто здесь. `[CITED: Reg.ru help/2domains — порт 587 + TLS вместо 25]`

**Грабли №3 — несуществующий ящик в `php.ini`/`From`.** Reg.ru требует, чтобы адрес отправителя реально существовал как ящик на домене хостинга. Значит: создать ящик (напр. `noreply@<домен>`) в панели Reg.ru и использовать его в `From`. `[CITED: Reg.ru help — «прописанный в php.ini ящик должен существовать»]`

**Грабли №4 — активная локальная почтовая служба при внешнем MX.** Если у домена MX смотрит на Яндекс, но в панели Reg.ru включена «локальная почтовая служба» — почта может зацикливаться/теряться. Это для адреса-ПОЛУЧАТЕЛЯ на стороне владельца, не для отправки; на доставку именно нашего письма влияет редко, но при разборе проблем проверить. `[CITED: Reg.ru help — снять «Активировать почтовую службу» при внешней почте]`

**Обязательные заголовки письма (UTF-8, кириллица):**
```php
// Source: общая практика PHP mail() + Reg.ru рекомендации (CITED)
$to      = 'MultiCrete@yandex.ru';
$subject = '=?UTF-8?B?'.base64_encode('Заявка с калькулятора — МультиКрит').'?='; // кодируем тему!
$headers   = [];
$headers[] = 'From: МультиКрит сайт <noreply@мультикрит.рф>'; // СВОЙ домен
$headers[] = 'Reply-To: '.$cleanEmail;                        // e-mail клиента (очищенный)
$headers[] = 'MIME-Version: 1.0';
$headers[] = 'Content-Type: text/plain; charset=UTF-8';
$headers[] = 'Content-Transfer-Encoding: 8bit';
// тело письма — тоже UTF-8
mail($to, $subject, $body, implode("\r\n", $headers));
```
- Тему с кириллицей **обязательно** кодировать (`=?UTF-8?B?...?=`), иначе «кракозябры» или отбраковка.
- `Content-Type: text/plain; charset=UTF-8` — иначе тело в кириллице ломается.

### Защита PHP-обработчика (V5 Input Validation — обязательно)

**Header injection.** Поля клиента (особенно e-mail → попадает в `Reply-To`) могут содержать `\r\n` и дописать `Bcc:`/`To:` → рассылка спама. `filter_var(FILTER_VALIDATE_EMAIL)` НЕ достаточен (пропускает экранированные пробелы/кавычки). `[CITED: sonarsource «Why mail() is dangerous», thesitewizard]`

**Защита:**
```php
// Source: thesitewizard / sonarsource (CITED) — отвергать любые CR/LF в полях,
// идущих в заголовки
function rejectIfInjection($v) {
  if (preg_match('/[\r\n]/', $v)) { http_response_code(400); exit('bad input'); }
  return $v;
}
$email = rejectIfInjection(trim($_POST['email'] ?? ''));
$name  = rejectIfInjection(trim($_POST['name']  ?? ''));
if (!filter_var($email, FILTER_VALIDATE_EMAIL)) { /* мягкая ошибка, e-mail необязателен */ }
// телефон/организация/сообщение — в ТЕЛО письма (не в заголовки), но всё равно
// прогнать htmlspecialchars при показе и обрезать длину.
```
- Поля, идущие в **заголовки** (`Reply-To`): отвергать при `\r\n`.
- Поля, идущие в **тело**: можно мягче, но обрезать длину и при HTML-выводе экранировать.

**Honeypot (анти-спам без капчи):** скрытое поле, которое люди не видят, а боты заполняют → если заполнено, тихо отбросить.
```html
<input type="text" name="website" tabindex="-1" autocomplete="off" style="position:absolute;left:-9999px" aria-hidden="true">
```
```php
if (!empty($_POST['website'])) { http_response_code(200); exit('{"success":true}'); } // тихо игнорируем бота
```

**Переиспользуемость (D-03):** один эндпоинт `send-lead.php`, поле `source` («calculator», позже «callback» и т.д.) — попадает в тему/тело письма, чтобы владелец видел, откуда заявка.
```php
$source = preg_replace('/[^a-z0-9_-]/i', '', $_POST['source'] ?? 'unknown'); // белый список
```

**Контракт ответа (для фронта):** возвращать JSON `{"success":true}` / `{"success":false,"error":"..."}` с корректным `Content-Type: application/json`, чтобы UI показал подтверждение/ошибку (LEAD-02).

### Fallback: Web3Forms (D-02 — ТОЛЬКО если Reg.ru-почта не заработает)

| Параметр | Значение |
|----------|----------|
| Эндпоинт | `POST https://api.web3forms.com/submit` `[CITED: docs.web3forms.com]` |
| Обязательное поле | `access_key` (из дашборда Web3Forms; письма уходят на e-mail, привязанный к ключу) |
| Тема | hidden `subject` |
| Имя отправителя | hidden `from_name` |
| Анти-спам | скрытый чекбокс `name="botcheck"` (honeypot, встроенный) |
| Ответ | JSON `{"success":true,"message":"..."}` |
| Способ | работает и как обычный `<form action=... method=POST>`, и через `fetch` с `Content-Type: application/json` |
| Лимиты | бесплатный тариф есть; точный месячный лимит в публичной доке не указан (исторически ~250/мес — `[ASSUMED]`, проверить на сайте при подключении) |
| Минус как fallback | зарубежный приёмник → данные клиентов уходят за рубеж (хуже по 152-ФЗ, чем Reg.ru — потому и fallback, а не основной) |

**Как переключаться:** форма и её поля остаются те же; меняется только `action`/URL отправки и добавляется `access_key`+`botcheck`. Спроектировать `sendLead()` так, чтобы endpoint был одной константой — переключение в одну строку.

**Проверка перед сдачей (обязательно по D-02):** отправить реальную тестовую заявку и убедиться, что письмо ПРИШЛО на `MultiCrete@yandex.ru` (проверить и папку «Спам»). Это приёмочный критерий этапа.

### Полноэкранное прокручиваемое окно на телефоне
Окно калькулятора больше окна лендинга (поля + шапка + оговорки + форма) → на телефоне должно занимать почти весь экран и **прокручиваться внутри**, не дёргая фон.

```css
/* Source: css-tricks / dev.to viewport units / jayfreestone (CITED) */
.calc-modal-overlay {
  position: fixed; inset: 0;
  background: rgba(26,35,50,.6); backdrop-filter: blur(4px);
  display: none; z-index: 100;

}
.calc-modal-overlay.open { display: flex; }

.calc-modal {
  background: var(--steel);                 /* #212C3D — тёмная тема */
  border-radius: 16px;
  width: 100%; max-width: 560px;
  max-height: 90dvh;                        /* dvh — реальная высота с учётом панелей браузера */
  overflow-y: auto;                         /* прокрутка ТЕЛА окна */
  overscroll-behavior: contain;            /* прокрутка не «утекает» на фон */
  -webkit-overflow-scrolling: touch;
}

@media (max-width: 560px) {
  .calc-modal-overlay { padding: 0; }       /* окно во весь экран */
  .calc-modal {
    max-width: 100%;
    height: 100dvh; max-height: 100dvh;     /* lvh/dvh вместо vh: нет «хвоста» под адресной строкой iOS */
    border-radius: 0;
    /* безопасные зоны iOS (вырез/домашняя полоса) */

  }
}
```

**Блокировка прокрутки фона.** Уже есть приём в лендинге: `document.body.style.overflow = 'hidden'` в `openModal` (строка 932). Этого достаточно для большинства случаев. На iOS Safari возможна «протечка» — усилить через `overscroll-behavior: contain` на самом окне (выше) — без сложного JS со сменой `position:fixed` тела. `[CITED: jayfreestone / benfrain — iOS body scroll lock]`

**Закрытие и доступность (UI-06 + WCAG):**
- По крестику, по клику на фон (`handleOverlay`), по `Esc` — все три уже есть в коде лендинга, переиспользовать.
- При открытии — перевести фокус на первое поле/крестик; при закрытии — вернуть фокус на кнопку-вызов.
- Атрибуты: `role="dialog"`, `aria-modal="true"`, `aria-labelledby="<id заголовка окна>"`.
- Лёгкий фокус-трап (Tab/Shift+Tab по элементам окна) — желателен; реализуется ~10 строками vanilla JS, без библиотек. `[CITED: hidde.blog / uxpin focus trap]`

**Читаемость «шапки» и формы на узком экране:**
- «Шапка» сметы (строки `подпись — значение`) → на телефоне в один столбец, значения выровнять по правому краю, `font-variant-numeric: tabular-nums` (как в `.specs-table`).

## Common Pitfalls

### Pitfall 1: Письмо уходит в спам Яндекса
**Что:** заявки «отправляются» (success), но владелец их не видит.
**Почему:** `From` на `@yandex.ru` (спуфинг) / нет SPF-выравнивания / тема не закодирована.
**Как избежать:** `From` на собственном домене, Яндекс в `Reply-To`, тема `=?UTF-8?B?`, `Content-Type: text/plain; charset=UTF-8`. Проверить ПАПКУ СПАМ при тесте.
**Признаки:** тест-письмо в «Спам» или не приходит вовсе → переходить к PHPMailer/SMTP 587, затем к Web3Forms.

### Pitfall 2: Header injection через поле e-mail/имя
**Что:** обработчик становится открытым релеем для спама.
**Почему:** `\r\n` в пользовательском поле дописывает заголовки; `filter_var` не ловит.
**Как избежать:** `preg_match('/[\r\n]/', ...)` → отказ для любого поля, идущего в заголовки.
**Признаки:** в логах/письмах появляются чужие `Bcc`/`To`.

### Pitfall 3: ИТОГО или оговорки спрятаны за форму
**Что:** нарушение LEAD-01/UI-04/D-11 — цена/контекст должны быть бесплатны.
**Как избежать:** за форму прячем ТОЛЬКО построчную детализацию (D-10); «шапка»+ИТОГО+оговорки+классы — сразу.

### Pitfall 4: Фон прокручивается под открытым окном на телефоне
**Что:** при скролле формы прокручивается лендинг за окном (особенно iOS).
**Как избежать:** `body{overflow:hidden}` (есть) + `overscroll-behavior: contain` на окне + `dvh` вместо `vh`.

### Pitfall 5: Правка движка ради UI
**Что:** соблазн «подкрутить» `calc()`/`params.js` под отображение.
**Как избежать:** UI только форматирует. Любая нехватка поля решается в UI, не в движке. Иначе ломается `engine.selftest.js` и эталон.

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|------------------|
| V2 Authentication | no | Нет аутентификации (публичная форма) |
| V3 Session Management | no | Нет сессий |
| V4 Access Control | no | Нет защищённых ресурсов |
| V5 Input Validation | **yes** | PHP: отвергать CRLF в полях-заголовках (`preg_match`), `filter_var` для e-mail (в теле), белый список для `source`, обрезка длины, `htmlspecialchars` при выводе |
| V6 Cryptography | no | Секретов нет; PHP-обработчик не хранит ключей в коде (CLAUDE.md) |

### Known Threat Patterns
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Email header injection (CRLF в поле → Bcc/спам) | Tampering / Elevation | `preg_match('/[\r\n]/', $field)` → отказ для всех полей в заголовках |
| Спам-боты через форму | DoS / Spoofing | Honeypot-поле; тихо отбрасывать заполненное |
| XSS при выводе заявки (если письмо HTML или есть веб-просмотр) | Tampering | `htmlspecialchars` при выводе; письмо — `text/plain` |
| Открытый релей / абьюз mail() | Elevation | Жёсткая валидация + фиксированный `$to` (адрес получателя НЕ из пользовательского ввода) |
| Секрет в коде (CLAUDE.md) | Information Disclosure | Никаких токенов/паролей в коммите; проверка после написания |
