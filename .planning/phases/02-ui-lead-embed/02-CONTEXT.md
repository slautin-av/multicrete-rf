<domain>
## Граница этапа

Этап 2 — интерфейс калькулятора поверх готового движка этапа 1. Клиент с лендинга открывает
калькулятор **всплывающим окном**, вводит площадь / толщину / класс сложности, сразу видит «шапку»
сметы с ИТОГО и ₽/м², оговорками и расшифровкой классов; при желании оставляет необязательную заявку
(контакты), которая реально доходит до владельца. Всё в фирменном тёмном стиле и на телефоне.

**В границах:** UI калькулятора (всплывающее окно, поля, валидация, результат), встраивание модуля
`calculator/` и кнопка-вызов на лендинге, форма заявки с реальной отправкой + согласие 152-ФЗ +
страница политики.

**НЕ в границах (см. Deferred):** переделка/оживление двух существующих форм лендинга
(«Заказать звонок», «Получить прайс»), блок «представители в регионах», правки самого движка расчёта.
</domain>

## Зафиксированные решения (LOCKED)

### Отправка заявки (LEAD-02)
- **D-01:** Заявка отправляется через **собственный PHP-обработчик на хостинге владельца (Reg.ru)**,
  который шлёт письмо на `MultiCrete@yandex.ru`. Свой сервер арендовать не нужно — хостинг уже оплачен
  и поддерживает PHP/почту. Данные клиентов остаются на российском хостинге — чисто по 152-ФЗ.
- **D-02:** **Web3Forms — запасной вариант.** Если при выполнении вскроются проблемы с доставкой писем
  с Reg.ru (mail() капризничает, письма не доходят на Яндекс) — переключаемся на Web3Forms (зарубежный
  приёмник по ключу, форма остаётся нашей). Помощник по выполнению ОБЯЗАН проверить доставку реального
  письма перед сдачей.
- **D-03:** В этом этапе обработчик подключается **только к форме калькулятора**. Но писать его как
  переиспользуемый модуль (один эндпоинт, параметр «источник заявки»), чтобы позже подключить другие формы.
- Доступы к хостингу Reg.ru владелец пришлёт позже — нужны только на стадии заливки файлов, не при планировании.

### Политика обработки персональных данных (LEAD-03)
- **D-04:** Создать **отдельную страницу политики** по типовому шаблону 152-ФЗ (напр. `privacy/index.html`
  или `policy.html`). Галочка согласия в форме ссылается на неё; без согласия отправка недоступна.
- **D-05:** Реквизиты оператора (ООО «МультиКрит» / ИНН / юр. адрес) для подстановки в шаблон владелец
  даёт при выполнении. Текст на 80% типовой — берём проверенный шаблон, подставляем реквизиты.

### Открытие калькулятора (UI-01, INT-01, INT-02)
- **D-06:** Кнопка на лендинге открывает калькулятор **всплывающим окном (оверлей) поверх страницы** —
  клиент не уходит с лендинга и не теряет контекст. Соответствует требованию «всплывающее окно».
- **D-07:** Калькулятор собирается как **отдельный модуль `calculator/`** (чистый HTML/CSS/JS, без сборщика,
  по образцу `materials/`). Одно другому не мешает: модуль можно и встроить окном, и при желании позже
  дать прямой ссылкой как страницу.
- **D-08:** Переиспользовать **готовую механику модального окна лендинга** (`openModal`/`closeModal`,
  стиль `.modal`, оверлей) — не изобретать заново. Кнопку-вызов разместить заметно; существующую кнопку
  «Получить расчёт» (сейчас скроллит к `#price`) логично перенацелить на открытие калькулятора.

### Глубина результата — воронка заявок (UI-03, LEAD-01)
- **D-09:** По умолчанию показываем **«шапку» сметы** (общая стоимость работ; в т.ч. трудозатраты,
  вспом. материалы, накладные, плановые накопления, налоги, командировочные = 0; стоимость материалов;
  ИТОГО; ₽/м²) + **ИТОГО видно сразу, без заполнения контактов** (требование LEAD-01 соблюдено — цена
  бесплатна).
- **D-10:** **Полная построчная детализация** (каждая строка работ и каждый расходник из `breakdown`)
  раскрывается **только после того, как клиент заполнит контакты** (имя, телефон, e-mail) + согласие.
  Это воронка: прозрачность работает как награда и мягко мотивирует оставить заявку. Форма остаётся
  НЕОБЯЗАТЕЛЬНОЙ — за контакты дают не цену, а углублённую раскладку.
- **D-11:** Текстовые оговорки (деталь до 80 кг; 1 день / 1 специалист; без транспорта/командировок/оснастки)
  и расшифровка классов 1/2/3 показываются **сразу вместе с «шапкой»** — это честный контекст цены, НЕ
  «секретная» детализация, прятать за форму нельзя (UI-04).

### Валидация и адаптивность (UI-05, UI-06)
- **D-12:** Ввод проверяется на корректность (площадь/толщина — положительные числа, класс выбран);
  при ошибке — понятная подсказка, расчёт не ломается. Конкретику UX-проверки (inline/при сабмите) —
  на усмотрение помощника.

### Claude's Discretion (на усмотрение помощника)
- Точная вёрстка окна и раскладка «шапки»; способ адаптации большого окна под телефон.
- Технический способ встраивания модуля в оверлей (инлайн-разметка vs подгрузка) — выбрать при планировании.
- Момент и форма валидации (по ходу ввода или при отправке).
- Имена файлов в `calculator/` для UI (латиница, дефисы — по правилам проекта).
</decisions>

### Движок этапа 1 (то, что UI отрисовывает)
- `calculator/engine.js` — функция `calc(inputs, P)` возвращает объект `breakdown`. UI вызывает её и
  рисует поля. «Шапка»: `trudozatraty`, `vspomMaterialy`, `nakladnye`, `pribyl`, `nds`, `transport`/
  `komandirovki` (=0), `materialy`, `itogo`, `perM2`. Полная детализация (за контактами): построчные
  `stDrobestruika`, `stObespylivanie`, `stNanesenie`, `stElastomer`, расходники и т.д. Оговорки — массив `ogovorki`.
- `calculator/params.js` — блок параметров `PARAMS` (один источник истины), не дублировать в UI.
- `calculator/engine.selftest.js` — самопроверка движка (контрольные примеры эталона).
- `.planning/phases/01-calc-engine/01-CONTEXT.md` — зафиксированные решения по движку (входы/выход, оговорки).

### Требования и метод

## Существующий код — на что опираться

### Переиспользуемые активы
- **Механика модального окна** в `landing.html`: `openModal()`/`closeModal()`, `.modal`, `.modal-close`,
  затемнение-оверлей — берём как основу всплывающего окна калькулятора.
- **Стиль кнопок и согласия**: `.btn`/`.btn-primary`/`.btn-secondary`, `.pf-consent`, `.pf-submit` —
  готовые классы фирменного стиля.
- **Движок `calculator/engine.js`** — детерминированная `calc()`, UI только вызывает и рисует.
- **`materials/index.html`** — рабочий образец структуры статического модуля.

### Установленные паттерны
- Чистый статический стек: HTML/CSS/JS, без сборщика, без npm. Калькулятор остаётся в этом стеке.
- Тёмная тема `landing.html` через CSS-переменные `--blue`/`--pink`/`--steel` (не `base.css`).
- ES-модуль: `engine.js` экспортирует `calc()` через `export function` — UI подключает через `<script type="module">`.

### Точки интеграции
- Кнопка-вызов на `landing.html` → открытие окна калькулятора (новая или перенацеленная «Получить расчёт»).
- Форма заявки калькулятора → новый PHP-обработчик на Reg.ru (эндпоинт + почта владельца).
- Галочка согласия → ссылка на новую страницу политики.
</code_context>

## Конкретные идеи владельца

- **Воронка «детализация за контакты»** (D-10) — собственная идея владельца: «шапка» и ИТОГО бесплатно,
  а полную построчную раскладку показываем после ввода имени/телефона/e-mail. Прозрачность как награда.

  (Reg.ru) уже оплачен; решили использовать его, а не арендовать что-то ещё.
