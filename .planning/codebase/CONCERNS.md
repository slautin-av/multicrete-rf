# CONCERNS — Риски и технический долг

## Технический долг (landing.html, инлайн-стили, дублирование)

- `landing.html` — монолит 1004 строки. Весь CSS (~490 строк) — инлайн в `<style>` внутри `<head>`. Никакого внешнего CSS-файла для лендинга нет, хотя у проекта уже есть `assets/styles/base.css` и `assets/styles/catalog.css` для других страниц. Рефакторинг в отдельный файл облегчит поддержку и позволит кешировать стили браузером.
- **19 инлайн-атрибутов `style=`** разбросано по HTML-разметке лендинга (например, `style="margin-top:1.5rem"`, `style="text-align:center"`, `style="color:var(--muted)"` и т.д.). Это мешает централизованному контролю отступов и цветов.
- **Дублирование блока «ДО/ПОСЛЕ»:** смысловой блок «Что разрушает металл — и как эластомер МультиКрит это останавливает» с картинками `fig1.png`/`fig2.png` и пятью карточками воздействий присутствует в двух разных файлах с разными классами CSS:
  - `landing.html` — классы `.dp__*`, стили вшиты инлайн в `<style>`.
  - `index.html` — классы `.mc-armor__*`, стили вшиты инлайн в `<style>`.
  Контент почти идентичен, вёрстка и именование классов — разные. При правке нужно синхронно менять оба файла, что неизбежно приведёт к расхождению.
- `index.html` — ссылки в навигации и футере (`problems/`, `about/`, `cases/`, `knowledge/`) ведут на несуществующие разделы (страницы отсутствуют в репозитории). На активном домене это будут 404.
- `index.html` и `landing.html` — две разные «главные» страницы проекта с разными заголовками (`MultiCrete — промышленные полимерные покрытия` vs `МультиКрит — Эластомерная броня для промышленности`). Непонятно, какая из них реально сервируется по корню. Риск путаницы при деплое.
- Нет сборщика (Webpack, Vite и т.п.) и нет `package.json`. Это осознанный выбор (статика), но при росте проекта (калькулятор, кейсы) отсутствие бандлера усложнит управление зависимостями.

## Производительность (WebGL-дым)

- `assets/js/webgl-fluid.umd.js` — 28 731 байт (минифицированный бандл сторонней WebGL-библиотеки). Грузится синхронно в конце `<body>` без `defer`/`async`. Это сравнительно небольшой файл, но он запускает GPU-симуляцию на весь сайт сразу при загрузке — без проверки возможностей устройства.
- На мобильных устройствах и слабых GPU (встроенная графика, бюджетные телефоны) WebGL-симуляция с `SIM_RESOLUTION: 128`, `BLOOM: true`, `SHADING: true` может давать заметный нагрев и просадку FPS. Параметр `SIM_RESOLUTION` и `BLOOM` — наиболее нагружающие.
- Нет фолбэка: если WebGL недоступен (старый браузер, принудительное отключение), инициализация `WebGLFluid(...)` просто упадёт с исключением. Страница останется тёмной (фон `#1A2332` из CSS), но скрипт выбросит ошибку в консоль. Контент читаем, но опыт деградирован не контролируемо.
- Нет проверки `prefers-reduced-motion` для WebGL-дыма. CSS-переходы для `.dp__frame img` под `prefers-reduced-motion: reduce` отключены корректно, но сам дым продолжает работать.
- Три шрифта Google Fonts (`Oswald`, `Manrope`, `Tektur`) — внешние запросы при каждой загрузке. `<link rel="preconnect">` стоит, это хорошо. Но шрифты не хостятся локально, что означает зависимость от CDN и лишний RTT.
- Внешняя зависимость Lucide Icons (`https://unpkg.com/lucide@latest`) грузится без `integrity`-атрибута и без фиксации версии (`@latest`). При обновлении библиотеки иконки могут поменяться или пропасть.

## Безопасность / секреты

- Файл `.env` **существует** в рабочем дереве (34 байта, создан 2026-05-08). Проверка `git ls-files --cached` показывает, что `.env` **не закоммичен** — он корректно исключён через `.gitignore` (строка `.env`). Риска утечки через репозиторий нет.
- `.env.example` **закоммичен** и содержит только имена переменных без значений (`ANTHROPIC_API_KEY=sk-ant-...`, `TELEGRAM_BOT_TOKEN=`, `TELEGRAM_CHAT_ID=`). Это штатно — шаблон без секретов.
- В `landing.html` и `index.html` нет никаких API-ключей, токенов или паролей в коде. Проверено поиском по ключевым словам.
- Форма обратной связи (`submitForm`, `submitCall`) в `landing.html` **не отправляет данные никуда** — оба обработчика делают только `e.preventDefault()` и вызывают `alert()`. Это заглушка. Пользователь думает, что отправил заявку — ничего не отправлено. Критический функциональный баг с точки зрения бизнеса (потеря лидов), не угроза безопасности.
- При реализации Telegram-бота (из `.env.example` — `TELEGRAM_BOT_TOKEN`) важно убедиться, что токен не попадёт в клиентский JS. Весь обмен с Telegram-API должен идти через серверную функцию (serverless/proxy), не напрямую из браузера.

## Доступность и SEO

- `landing.html`: у трёх изображений есть `alt` (`Logo_Multicrete.png`, `fig1.png`, `fig2.png`, `equipment-collage.png`) — тексты описательные, не пустые. Это хорошо.
- Иконки Lucide рендерятся через `<i data-lucide="..."></i>` — после инициализации `lucide.createIcons()` превращаются в `<svg>`. SVG-иконки не имеют `aria-label` и не скрыты через `aria-hidden="true"`. Скринридер попытается читать пустые SVG-элементы.
- `landing.html` не содержит `<link rel="canonical">` и Open Graph-тегов (`og:title`, `og:description`, `og:image`). Для B2B-страницы, где SEO критичен, это упущение.
- Нет `favicon`: в `<head>` не подключена иконка сайта. Вкладка будет с дефолтным значком браузера.
- Семантика в целом корректна: `<header>`, `<section>`, `<footer>`, `<nav>`, `<main>` (в `index.html`) присутствуют. В `landing.html` тег `<main>` отсутствует — весь контент идёт прямо в `<body>`.
- Кнопки с `onclick="..."` в атрибутах (например, `onclick="openModal()"`, `onclick="goTo('#price')"`) работают, но нарушают принцип разделения контента и поведения. При минификации HTML это может создать проблемы.
- В `index.html` год в футере — `© 2025`, в `landing.html` — `© 2026`. Несогласованность.

## Риски будущего портала (из ВИДЕНИЯ-ПОРТАЛА.md)

- **Калькулятор — приватность модели ценообразования.** Коэффициенты сложности — ноу-хау. При реализации на статическом сайте (HTML/JS) весь код калькулятора окажется в браузере. Любой конкурент откроет DevTools и прочитает формулы. **Решение, зафиксированное в ВИДЕНИИ:** публичный калькулятор отдаёт только «вилку» (диапазон), точный движок с коэффициентами работает только в режиме Алексея — и этот режим должен быть серверным (serverless function, backend) или изолированным (локальный инструмент, не публикуется). Нельзя класть реальные коэффициенты в клиентский JS — даже в минифицированном виде они тривиально читаются.
- **Хранилище кейсов — выбор формата.** Сейчас репозиторий — чистая статика без CMS. Для кейсов придётся выбрать: JSON-файлы в репо (просто, но правка через git), Notion/Airtable как headless CMS (удобно Алексею, но внешняя зависимость), или локальная БД (SQLite, если появится серверная часть). Решение влияет на архитектуру — его нужно принять до начала стройки.
- **Автопубликация кейсов** (голос + фото → сайт) требует минимум: хранилище с API, CI/CD или webhook для пересборки/обновления страниц. Статический сайт без сборки не умеет «принять» новый кейс без ручного деплоя.
- **Telegram-бот** (токен уже в `.env.example`) планируется как канал уведомлений. Пока форма — заглушка (см. Безопасность). Связь формы → Telegram должна идти через серверную функцию, не напрямую из JS клиента.

## Фрагильные места (что легко сломать при доработке)

- **Один большой CSS-блок в `<head>` `landing.html`** (~490 строк). Любая правка цветовой переменной или базового компонента затрагивает весь файл без изоляции. Нет CSS-модулей, нет слоёв (`@layer`).
- **Дублированный блок ДО/ПОСЛЕ** в `landing.html` и `index.html` с разными классами. Правка текста или картинок в одном файле не отразится в другом. При доработке лендинга эти блоки будут дрейфовать.
- **WebGL-инициализация без try/catch** (`WebGLFluid(fluidCanvas, {...})`). Если библиотека не загрузится (нет сети, CDN Lucide упал), строка на 967 упадёт с `ReferenceError: WebGLFluid is not defined`. Нет обёртки, нет деградации.
- **Lucide `@latest` без integrity.** Если `unpkg.com/lucide@latest` сменит API иконок (переименование), все иконки исчезнут одновременно на всех страницах где используется Lucide.
- **Навигация `landing.html`** ссылается на якоря (`#spheres`, `#portfolio`, `#specs`, `#price`, `#contacts`). Якоря есть в HTML, но если секцию переименуют или удалят — ссылки в nav молча перестанут работать (нет автоматической проверки).
- **Изображение `equipment-collage.png`** в `assets/images/sections/` отмечено в `git status` как untracked (`??`). Оно используется в `landing.html` (строка 699), но ещё не добавлено в репозиторий. При деплое с чистого клона изображение будет отсутствовать — секция оборудования покажет битую картинку.
