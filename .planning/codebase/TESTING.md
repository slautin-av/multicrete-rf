# TESTING — Тестирование

## Текущее состояние

Автоматических тестов нет. В проекте отсутствуют:
- файлы `*.test.js`, `*.spec.js`, `*.test.ts`
- директории `tests/`, `__tests__/`, `spec/`
- конфигурации тестовых фреймворков (`jest.config.js`, `vitest.config.js`, `.mocharc.*`)
- `package.json` (нет Node.js-инфраструктуры вообще)
- CI/CD-пайплайны (`.github/workflows/`, `gitlab-ci.yml`)

Проект — статический HTML/CSS/JS без сборщика. Никакой автоматизированной проверки качества не настроено.

---

## Как сейчас проверяется результат

Проверка — визуальная, вручную в браузере. Для этого в проекте есть отдельные демо-страницы:

- `smoke-demo.html` — изолированный стенд для проверки эффекта интерактивного дыма (WebGL fluid). Позволяет смотреть на поведение дыма в фирменных цветах без остального контента сайта.
- `do-posle-demo.html` — изолированный стенд для блока «ДО / ПОСЛЕ» с эластомерной бронёй. Тестирует вёрстку секции сравнения в отрыве от лендинга.
- `_research/design-refs/claude-design-итерация-3-финальная.html` — финальная итерация дизайна карточки материала (артефакт-эталон по `DESIGN.md`), используется как эталон при вёрстке каталога.

Таким образом, итерационная работа ведётся через создание изолированных HTML-стендов, которые затем интегрируются в основные страницы (`landing.html`, `index.html`).

## Рекомендации на будущее

**Текущий сайт** (статические страницы без логики) в тестировании через фреймворки не нуждается — визуальной проверки достаточно.

**При добавлении калькулятора** (вероятная следующая фича — подбор материала или расчёт расхода) логику стоит вынести в отдельный JS-модуль и покрыть юнит-тестами:

```
assets/js/calculator.js       ← чистая функция, без DOM
assets/js/calculator.test.js  ← тесты через Vitest или Jest
```

Минимальный набор проверок для калькулятора:
- корректный расчёт при граничных значениях (0, максимум, дробные числа)
- обработка некорректного ввода (пустая строка, отрицательное число, нечисло)
- соответствие результата ожидаемому для типовых задач (площадь → расход → число упаковок)

**Визуальная регрессия** при масштабировании сайта (добавление новых страниц материалов) — можно покрыть скриншот-тестами через Playwright или Puppeteer, запуская их перед каждым деплоем. Пока преждевременно.
